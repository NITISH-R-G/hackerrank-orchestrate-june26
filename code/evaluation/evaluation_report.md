### Projected full test set (`dataset/claims.csv`, 44 rows)

- **Model calls (est):** ~44 (winner = single-pass)

- **Tokens (est):** ~5865 input + ~719 output

- **Images processed (sample):** 31; **(test):** 85

- **Cost (est, USD):** ~$0.004 at $0.3/M input, $2.5/M output (gemini-2.5-flash)

- **Latency (est):** ~13s/row throttled => ~10 min for 44 rows


### Rate-limit / retry / caching strategy

- **Rate limit:** free tier is **5 RPM** for gemini-2.5-flash (confirmed by a 429 during dry-run). Enforced via `_throttle()` sleeping `SLEEP_BETWEEN_CALLS=13s` before every call, keeping us under 5 RPM.

- **Retry:** `tenacity`-style loop, MAX_RETRIES=5, exponential backoff (multiplier 2, min 13s, max 60s) on 429/5xx only; non-retryable (400/401) propagate immediately.

- **Caching:** deterministic on-disk cache keyed on (model, temperature, prompts, image SHA-256 hashes). Cache hit = 0 model calls, 0 tokens. Enables resume-after-crash (US-006) without reprocessing completed claims.

- **Batching:** none (sequential), because the 5 RPM ceiling makes concurrency counterproductive.


## 7. Recommendations for production deployment

- Move to a paid tier to raise the RPM ceiling; then add modest concurrency (2-3 workers) with a token-bucket limiter.

- Keep the deterministic normalization + consistency layers as the contract guarantor regardless of model.

- Add per-user caching of the Pass 1 intent extraction when the same conversation is re-submitted.

- Monitor `pass1_failures`/`pass2_failures`; sustained failures should page an operator (indicates model outage or quota).
